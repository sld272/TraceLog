# TraceLog 参赛开发方向、方案与时间线

本文档基于当前仓库状态与两场比赛要求整理，用于指导 TraceLog 先完善产品、报名参加“中国移动杯”2026江苏生成式人工智能创新大赛，再进一步面向大学生场景优化参加 2026 EL 智能应用开发与创新大赛交互组，最后扩展 Agent 版本参加 EL 大赛 AI 智能体创新专项组。

## 1. 当前项目状态判断

### 1.1 项目定位

TraceLog 是一款面向普通学生与青年用户的陪伴型 AI 产品，由两条核心支柱构成，并以递进的方式形成完整产品逻辑：

- **入口层：向内的 AI 社交媒体**
  用最熟悉的“发帖”交互承接用户日常表达，让记录像发朋友圈一样轻。AI 是用户唯一的读者，解决“愿不愿意写”的问题。
- **价值层：AI 成长记忆引擎**
  把每一次轻量输入沉淀为长期可用的画像、记忆、待办与复盘，解决“写了有没有用”的问题。

一句话叙事：

> TraceLog 用社交媒体的轻量交互承接表达，用成长记忆引擎沉淀价值，让普通人也能拥有一个长期理解自己、陪伴自己成长的 AI。

需要特别强调“向内运行的社交媒体”这一概念，避免被误解为多人社交产品。对外推荐统一表述：

- 向内的社交媒体
- 单人社交媒体
- 给自己发的朋友圈
- AI 是你唯一的读者

#### 与专业型 Agent 的赛道区分

TraceLog 与 Claude Code、Cursor、OpenHands 等专业型 Agent 不在同一赛道：

| 维度 | 专业型 Agent | TraceLog |
| --- | --- | --- |
| 用户定位 | 开发者、专业工作者 | 普通学生与青年用户 |
| 核心价值 | 提升专业生产力 | 陪伴个人生活与成长 |
| 交互形态 | 任务指令型 | 表达式、社交化、轻量化 |
| 数据沉淀 | 项目代码与工作产出 | 长期个人记忆与成长轨迹 |

这是当前 AI 产品中尚未被巨头吃下的清晰空位，对江苏 AIGC 评委（产业空间）和 EL 评委（学生共鸣）都更有故事感。

### 1.2 已实现能力

当前仓库已经具备以下核心能力：

- **CLI 交互入口**：用户通过命令行输入日常记录，系统返回温暖、有洞察力的中文回应。
- **本地 Markdown 记忆存储**：每条用户输入会保存为独立 Markdown 帖子，带有 YAML frontmatter。
- **用户画像沉淀**：退出时会读取旧画像和近期帖子，由 LLM 整理生成结构化 `profile.md`。
- **待办提取与管理**：LLM 从用户明确表达的任务中提取待办，写入 `todos.json`。
- **语义检索记忆**：使用 ChromaDB 和 OpenAI embedding，对历史帖子建立向量索引，并在新输入时检索相关历史。
- **Markdown + JSON 双引擎数据结构**：Markdown 负责人类可读与长期保存，JSON 负责结构化任务管理。
- **可配置模型与 embedding**：首次运行可配置主模型、base URL、embedding 模型和 embedding 服务。

### 1.3 当前短板

从参赛角度看，当前项目仍处在“技术原型”阶段，主要短板是：

- **没有前端界面**：目前只有 CLI，交互展示力不足，不利于评委快速理解产品。
- **没有稳定演示路径**：缺少预置 demo 数据、演示脚本、截图和可复现流程。
- **个人成长闭环不完整**：已有记录、记忆、画像、待办，但缺少回顾、洞察、趋势、目标追踪等成长反馈。
- **大学生场景不足**：当前定位偏通用个人成长，尚未针对课程、DDL、社团、考试、科研、竞赛等大学生活做专门优化。
- **Agent 能力还不突出**：目前是“AI 回复 + 信息抽取 + 检索”，还不是能主动规划、调用工具、生成周报、追踪目标的智能体。
- **工程完整度不足**：缺少测试、Web API、部署说明、错误状态展示、数据导入导出、隐私说明等。

### 1.4 适合的总方向

TraceLog 的总方向已经清晰：

> 面向普通学生与青年用户的陪伴型 AI 产品，由“向内的 AI 社交媒体”作为入口，由“AI 成长记忆引擎”作为价值底座。

同一套底层产品，配合不同的“放大镜”，可以自然适配三场参赛目标：

1. **江苏 AIGC 大赛**：双引擎都讲，但主推“成长记忆引擎”作为 AIGC 创新点（长期记忆、画像生成、语义检索、自动复盘），把“向内的社交媒体”作为交互创新呈现。强调产业空间和落地潜力。
2. **EL 交互组**：主推“AI 社交媒体”叙事，因为交互组评的是体验和创意，学生评委对“给自己发朋友圈”这种比喻最有共鸣。
3. **EL Agent 组**：主推“成长记忆引擎”作为 Agent 的数据底座，Agent 是引擎之上能主动行动的层，强调智能体规划、工具调用与平台能力。

## 2. 总体产品方向

### 2.1 产品名称与一句话介绍

建议继续使用：

- 中文名：拾迹
- 英文名：TraceLog
- Slogan：致敬每一份不被遗忘的成长

通用一句话介绍（用作 PPT 封面与项目简介首句）：

> TraceLog 拾迹是一款面向普通学生与青年用户的陪伴型 AI 产品。它用社交媒体的轻量交互承接日常表达，用成长记忆引擎沉淀长期画像、记忆与行动，让普通人也能拥有一个长期理解自己、陪伴自己成长的 AI。

不同比赛的强调侧重见 [2.4 节](#24-不同比赛的叙事侧重)。

### 2.2 核心价值主张：双引擎递进结构

TraceLog 的价值不是“两个并列功能”，而是“一条递进路径”。两条支柱有明确分工：

#### 入口层：向内的 AI 社交媒体（解决“愿不愿意写”）

- **熟悉的交互**：像发朋友圈一样发帖，0 学习成本。
- **轻量的输入**：一句话、一段话、一段心情都可以。
- **温暖的回应**：AI 作为唯一读者给出共情回复，让记录有反馈。
- **私密的空间**：写给自己看，不必担心社交压力或表达完美度。

这一层解决传统日记和笔记产品最大的难题：用户写不下去、坚持不下来。

#### 价值层：AI 成长记忆引擎（解决“写了有没有用”）

- **长期记忆**：基于向量检索的语义记忆系统，让历史不再沉没。
- **画像沉淀**：AI 自动整理用户的身份、技能、兴趣、习惯、目标和情绪倾向。
- **行动转化**：从自然语言中提取待办、DDL 和计划，减少遗漏。
- **阶段复盘**：自动生成周报/月报、关键词、趋势和成长建议。
- **数据主权**：本地 Markdown + JSON 存储，用户能看见、迁移和掌控自己的数据。

这一层让“随手写下的碎片”转化为“能被长期复用的成长资产”，是 TraceLog 真正的 AIGC 创新所在。

#### 两层之间的因果关系

- 入口层让用户**愿意持续输入**，否则记忆引擎没有数据。
- 价值层让用户**得到持续回报**，否则用户不会持续输入。

这一闭环是 TraceLog 区别于普通聊天机器人和普通笔记 App 的核心。

### 2.3 目标用户：分层叙事（江苏 AIGC 宽 / EL 窄）

底层产品和代码是同一个，**只是对外讲故事时切换不同的“放大镜”**。

#### 江苏 AIGC 大赛：用户群体放宽，但要“宽得有结构”

江苏 AIGC 评委关注产业价值和落地潜力，因此目标用户应该宽，但必须**有清晰层次**，避免讲成“所有人都能用”那种没有锚点的产品。建议讲成同心圆结构：

- **核心用户（验证切入点）**：学生与初入职场的青年群体。理由是这一群体表达欲强、记录需求高、成长节奏明显，最适合做产品早期验证。
- **扩展用户（市场空间）**：所有需要长期个人记忆与自我管理的普通人，如自由职业者、新手父母、终身学习者。
- **远期场景（产业潜力）**：可延伸至企业员工成长档案、心理健康辅助、终身学习记录、家庭记忆共享等。

这样既能体现产业空间，又能给评委一条清晰的落地路径：

> 我们选择从大学生群体切入做产品验证，因为这个群体表达欲强、记录需求高、成长节奏明显；验证成功后向青年群体和职场新人扩展，远期可服务所有需要长期个人记忆的普通人。

#### EL 大赛：用户群体收紧到大学生，并具体化画像

EL 评委本身就是南大学生和老师，对“真不真实”的判断非常敏锐。目标用户不只是“大学生”，而要收紧到**有画像的子群体**，让评委一眼就觉得“这是我”。建议从以下方向中选择 1—2 个作为核心锚点：

- 想认真生活但容易半途而废的大学生。
- 同时被课程、社团、竞赛、DDL 挤压的多线程学生。
- 爱表达但不爱整理的人文社科学生。
- 大一新生：环境陡变、需要外置大脑。

锚点越具体，demo 故事越能打动评委。EL 演示数据集应围绕一个具体的虚构用户构建，例如“小南，南大 2025 级本科生，理科，社团积极分子，正在准备 EL 比赛”。

#### 两套叙事的对应关系

| 维度 | 江苏 AIGC 版叙事 | EL 版叙事 |
| --- | --- | --- |
| 目标用户页 | 同心圆：核心是学生青年，扩展到普通人 | 单一画像：具体的大学生子群体 |
| 痛点页 | 普通人长期记忆缺失、个人成长难追踪 | 大学生多线程压力、记录易半途而废 |
| Demo 数据 | 学生 + 职场新人混合 | 纯校园场景 |
| 评委关注点 | 产业空间、落地路径、AIGC 创新 | 真实需求、产品体验、学生共鸣 |

### 2.4 不同比赛的叙事侧重

同一产品在三场比赛中的“放大镜”切换：

| 比赛 | 主推支柱 | 副支柱 | 关键词 |
| --- | --- | --- | --- |
| 江苏 AIGC 大赛 | AI 成长记忆引擎 | 向内的 AI 社交媒体（作为交互创新） | 长期记忆、画像、复盘、产业空间、数据主权 |
| EL 交互组 | 向内的 AI 社交媒体 | AI 成长记忆引擎（作为体验深度） | 校园场景、交互创意、产品体验、学生共鸣 |
| EL Agent 组 | AI 成长记忆引擎（作为 Agent 数据底座） | Agent 主动行动层 | 工具调用、任务规划、自动复盘、Coze 平台 |

### 2.5 典型使用场景

建议围绕这些场景设计 demo（江苏 AIGC 版本可适当混合非校园场景，EL 版本应纯校园）：

1. **日常碎片记录**：用户输入“今天高数作业拖到了晚上，感觉效率很低，但下午和队友讨论了比赛想法”。
2. **自动待办提取**：系统识别“周五前交项目 PPT”“明天下午开组会”等任务。
3. **历史语义回忆**：用户问“我最近为什么总觉得累？”系统检索近期记录，结合画像给出洞察。
4. **阶段复盘**：系统生成本周学习、情绪、社交、项目进展总结。
5. **成长建议**：系统根据长期画像给出下一周行动建议。
6. **校园竞赛管理**：记录比赛想法、DDL、队伍分工、材料准备，自动形成参赛计划。

## 3. 推荐技术路线

### 3.1 记忆系统重构（v3 架构）

记忆系统是 TraceLog 真正的 AIGC 创新点。详细技术设计见 [`memory_architecture.md`](./memory_architecture.md)，本节只列出影响参赛叙事的关键决策：

**分层结构**

| 层级 | 存储 | 进 system prompt | 作用 |
| --- | --- | --- | --- |
| L1 人格 | `souls/*.md` 文件库 | ✓（当前激活的一个） | AI 对话语气与边界 |
| L2 用户档案+成长画像 | `user.md`（章节合并） | ✓ | 谁、在变成谁 |
| L3 结构化记忆 | `state.db` (SQLite) | ✗ | 帖子、实体、情绪、事件、待办、反思 |
| 语义索引 | `chroma_db/` | ✗ | 向量相似度检索 |

**关键技术选型**

- **SQLite + FTS5 双 tokenizer**：unicode61 + trigram，原因是中文用户必须用 trigram 才能子串匹配（参考 Hermes Agent 源码）。
- **ChromaDB 保留**：FTS5 解决精确，ChromaDB 解决语义。两者通过 RRF 融合做混合检索。
- **反思器异步触发**：每条 post 后台 spawn 轻量 LLM agent 抽取实体/情绪/事件；周/月深反思更新画像（参考 Hermes `background_review.py`）。
- **数据主权 v2**：所有数据本地，前端做编辑入口，提供 `tracelog export` 一键导出为 Markdown 包。

**与 Hermes Agent 的差异化**

| 维度 | Hermes（参考） | TraceLog |
| --- | --- | --- |
| 用户群 | 开发者 | 普通学生青年 |
| 语义检索 | 外挂 provider | 内置 ChromaDB |
| 容量限制 | 字符硬限 | 软限，前端可视 |
| L2 文件 | USER + MEMORY 两个 | user.md 一个，章节区分 |

### 3.2 当前架构演进方向

建议从当前 CLI 原型演进为三层结构：

1. **Core Layer：核心记忆与智能逻辑**
   - 当前 `memory.py`、`router.py`、`vectorstore.py` 的能力被新架构吸收。
   - 业务流程抽成可被 Web/API/Agent 复用的服务函数。
2. **API Layer：后端服务**
   - 使用 FastAPI 暴露记录、聊天、待办、画像、复盘、搜索、SOUL 切换等接口。
   - 为前端和 Agent 版本提供统一能力。
3. **UI / Agent Layer：展示与智能体入口**
   - Web 前端用于江苏 AIGC 和 EL 交互组展示。
   - Coze 把 API 包装为 Agent tools，用于 EL Agent 组展示。

### 3.3 建议新增模块

```
core/
├── db.py                  # SQLite 连接 + WAL + 重试封装
├── record_service.py      # 发帖：写 SQLite + ChromaDB
├── context_builder.py     # 组装 system prompt + 检索上下文
├── retrieval.py           # FTS5 + ChromaDB + RRF
├── reflector.py           # 轻反思 / 深反思
├── soul_service.py        # SOUL 切换、加载、列表
├── profile_service.py     # user.md 章节级 patch
├── todo_service.py        # 待办 CRUD
└── export_service.py      # tracelog export 实现
api/                       # FastAPI 接口（第二期）
frontend/                  # Web 前端（第二期）
agent/                     # Coze tools 适配（第三期）
migrations/
└── 001_init.sql           # SQLite schema
scripts/
└── migrate_to_v3.py       # 一次性数据迁移
demo/                      # 演示数据 + 演示脚本
tests/
```

不建议一开始大重构，先按 [`memory_architecture.md`](./memory_architecture.md) §10 的"第一期清单"推进，保证 5/31 前演示可用。

### 3.4 Web 前端方向

前端建议做成"个人成长时间线 + AI 侧边栏"的形态：

- 首页：今日记录输入框、AI 回复、最近记录。
- 时间线页：按日期展示历史帖子。
- 待办页：展示 AI 提取的任务、DDL、完成状态。
- 画像页：展示 `user.md` 中 AI_PROFILE 段，支持手动编辑 USER_FACTS 段。
- SOUL 切换页：当前激活的 SOUL、SOUL 库列表、新建 SOUL。
- 复盘页：展示周报/月报、情绪曲线、关键词、关系图、行动建议。
- 搜索页：双轨检索历史记录。

视觉风格建议：温暖、安静、类社交媒体，但强调"只给自己看"。

### 3.5 Agent 版本方向

Agent 版本不应只是把聊天搬到 Coze，而应突出"主动规划和工具调用"。建议定义为：

> TraceLog Growth Agent：一个能读取个人记忆、分析近期状态、拆解目标、追踪 DDL、生成复盘与行动计划的成长智能体。

Agent 工具列表（详见 `memory_architecture.md` §10.3）：

- `search_memory(query, mode)`：双轨检索历史
- `query_entity(name, type)`：查特定人物/课程/项目的所有相关记录
- `get_emotion_trend(days)`：情绪趋势分析
- `list_todos / add_todo / update_todo`：待办管理
- `generate_reflection(scope)`：触发深反思
- `switch_soul(name)`：切换 AI 人格
- `add_post(content)`：Agent 也能帮用户记录

如果使用 Coze，可以把 TraceLog 后端接口包装为插件/API；如果来不及，也可以先做一个 Coze 原型，使用模拟数据和固定工作流展示 Agent 能力。

## 4. 三阶段参赛策略

## 阶段一：完善项目，报名江苏 AIGC 大赛

### 4.1 参赛定位

江苏 AIGC 大赛建议优先选择：

1. **AIGC创新产品赛道**：最稳妥。TraceLog 可以作为面向个人成长和知识管理的 AIGC 产品。
2. **智能体赛道**：如果在报名材料中突出长期记忆、任务提取、主动复盘和目标规划，也可以包装为个人成长智能体。
3. **OPC创新创业子赛道**：如果强调个人开发者、小团队、轻量化创业闭环，可作为备选叙事。

推荐主叙事：

> TraceLog 是一款具备长期记忆能力的个人成长 AIGC 产品，帮助用户把碎片化表达转化为结构化记忆、行动计划和自我洞察。

### 4.2 江苏 AIGC 阶段必须完成的产品能力

按优先级排序：

#### P0：必须完成

- 稳定运行当前 CLI 核心流程。
- 修复明显 bug 和异常体验。
- 准备一组高质量 demo 数据。
- 补充 README：项目介绍、安装、配置、运行、功能说明、架构说明。
- 生成可演示的用户画像、帖子、待办和语义检索效果。
- 录制基础演示视频。
- 准备报名用项目介绍和创新点。

#### P1：强烈建议完成

## 阶段三：Agent 版本，参加 EL AI 智能体创新专项组

### 6.1 Agent 组定位

EL Agent 组要求聚焦智慧校园、智慧生活，并使用火山引擎·扣子（Coze）作为智能体开发工具。TraceLog Agent 版本建议定位为：

> 面向大学生的成长规划 Agent，可以基于个人记录和目标，自动生成复盘、拆解计划、追踪 DDL，并给出下一步行动建议。

### 6.2 Agent 版本与交互组版本的区别

交互组强调“产品界面和用户体验”；Agent 组强调“智能体工作流和任务完成”。

Agent 版本应该回答：

- 我能帮你规划什么？
- 我能调用哪些工具？
- 我能记住什么？
- 我能如何持续追踪？
- 我能怎样把记录转化为行动？

### 6.3 Agent 核心能力设计

建议设计 5 个 Agent 技能：

1. **成长复盘 Agent**
   - 输入：最近一周记录。
   - 输出：学习、生活、情绪、社交、项目总结。
2. **DDL 管理 Agent**
   - 输入：自然语言记录。
   - 行动：提取待办、设置截止时间、提醒优先级。
3. **目标拆解 Agent**
   - 输入：目标，例如“我要准备 EL 答辩”。
   - 输出：阶段计划、任务清单、时间安排。
4. **记忆检索 Agent**
   - 输入：用户问题，例如“我之前为什么想做这个项目？”
   - 行动：检索历史记录，给出引用式回答。
5. **状态关怀 Agent**
   - 输入：近期情绪记录。
   - 输出：温和反馈、风险提示、可执行建议。

### 6.4 Coze 落地方式

如果时间充足：

- 使用 TraceLog 后端提供 API。
- 在 Coze 中配置插件/API：
  - `add_record`
  - `search_memory`
  - `list_todos`
  - `update_todo`
  - `generate_weekly_review`
  - `get_profile`
- Coze Agent 调用这些接口完成任务。

如果时间紧张：

- 先在 Coze 中构建纯工作流版本。
- 使用固定示例数据模拟用户记忆。
- 展示 Agent 如何从记录中生成复盘、计划和建议。
- 后续再接入真实后端 API。

- 智能体体验链接或部署包。
- 项目文档。
- 演示视频。
- 源码或配置说明。

## 12. 风险与应对

### 12.1 时间风险

风险：江苏 AIGC 报名截止较近，Web 前端可能来不及完成。

应对：先以 CLI + 演示视频 + 清晰材料报名，把 Web 前端放到 6 月完成。

### 12.2 演示风险

风险：现场或录屏时模型 API、embedding 或网络异常。

应对：准备固定 demo 数据、截图、备用视频；演示前预先生成 profile、todos、posts 和检索结果。

### 12.3 定位风险

风险：项目容易被理解为“AI 日记”或“聊天机器人”，显得普通。

应对：所有材料都强调“长期记忆 + 成长闭环 + 本地数据主权 + 行动转化”。

### 12.4 Agent 组平台风险

风险：Coze 平台接入后端 API 或部署存在不确定性。

应对：准备两套方案：真实 API 接入为主，Coze 模拟工作流为备选。

### 12.5 合规风险

风险：个人记录可能涉及隐私，AI 建议可能被误解为心理咨询或专业建议。

应对：明确产品定位为成长记录与自我管理工具，不提供医疗、心理诊断、法律或金融等专业结论；强调本地存储和用户数据自主权。

## 13. 最终建议

TraceLog 当前最值得保留和放大的核心，不是“聊天”，而是“长期记忆”。比赛开发应围绕一个主线推进：

> 让 AI 从一次性回复，变成能持续理解、整理和陪伴用户成长的个人记忆智能体。

因此，短期不要分散精力做太多花哨功能。优先把“记录—理解—记忆—待办—复盘—计划”这条闭环做完整、讲清楚、演示稳定。这样同一个项目可以自然适配江苏 AIGC 的产品创新叙事、EL 交互组的校园体验叙事，以及 EL Agent 组的智能体能力叙事。
